# M9: Init UX Improvements

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**Before coding, run:**
- `/create-adr` for auto-detection design decisions (scope, API, vitest-over-jest priority)
- `/create-adr` for non-interactive init (scope change from v2, flag semantics, TTY policy)
- `/create-user-scenario` for all scenarios listed in the [plans](#specifications)

**MUST implement:**
- New `cli/internal/detect` package: `Scan(dir string) Result` (never returns error)
- Detection from `package.json` (`dependencies` + `devDependencies`) and `Gemfile`
- Modified `quarantine init` prompt: single detected (default in brackets), multiple
  (numbered list + name input), none (current behavior unchanged)
- Flags on `quarantine init`: `--yes`/`-y`, `--framework`, `--retries`, `--junitxml`
- `--yes` mode: skips all prompts, requires `--framework`, overwrites config with log
- Flag overrides without `--yes`: each flag skips exactly its own prompt
- Retries range validation (1–10) in both interactive and non-interactive paths
- TTY warning when stdin is not a terminal and `--yes` is absent

**MUST NOT implement:**
- Package manager detection (no consumer in v1)
- JUnit XML path detection from existing config files
- `--config` flag on `quarantine init`
- Changes to `quarantine run`, `quarantine doctor`, or `quarantine version`

## Acceptance criteria

From [index.md](index.md#m9-init-ux-improvements):

1. `quarantine init` detects jest/vitest from `package.json` as prompt default (FR-1.4.2)
2. `quarantine init` detects rspec from `Gemfile` as prompt default (FR-1.4.2)
3. Multiple detected frameworks: numbered list, vitest before jest, selectable by number or name
4. No detection → current prompt behavior, unchanged
5. Malformed `package.json` → silent fallback to undetected behavior
6. `quarantine init --yes --framework jest` creates config without prompts, exits 0
7. `quarantine init --yes` without `--framework` exits 2 with actionable error
8. `quarantine init --framework jest` (no `--yes`) skips framework prompt only
9. Invalid `--framework` exits 2 immediately, no fallthrough to interactive prompt
10. `quarantine init --yes --framework jest --retries 11` exits 2
11. Interactive retries prompt rejects out-of-range values and re-prompts
12. `make cli-build && make cli-test && make cli-lint` pass

Requirements: [functional](../specs/functional-requirements.md),
[non-functional](../specs/non-functional-requirements.md).

## Verification

**Build:** `make cli-build && make cli-test && make cli-lint` MUST pass.

**Detect package:** `detect.Scan()` MUST never return an error (advisory-only API).
Detection failures MUST silently fall back to Case C (no-detection prompt).

**Non-interactive:** `quarantine init --yes` without `--framework` MUST exit 2.
Flag overrides MUST skip exactly the flagged prompts and no others.

**Retries:** Both interactive and non-interactive paths MUST reject values outside
1–10 with a clear error or re-prompt.

**Requirements:** Implementation MUST satisfy [FR-1.4.2](../specs/functional-requirements.md).
