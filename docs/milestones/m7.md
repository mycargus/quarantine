# M7: Dashboard Analytics and UI

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**MUST implement:**
- Quarantined test list per project: test name, issue link, quarantine date,
  flaky count
- Trend charts: flaky test count over time, test stability trend (pass rate),
  quarantine queue size over time
- Filters: by project, date range, test status
- Search: by test name or `test_id`
- Cross-repo overview page: aggregated stats (total quarantined, total flaky
  detections, most flaky tests)
- On-demand sync on page load: check debounce, pull artifacts from GitHub,
  ingest into both `test_runs` and `quarantined_tests` (FR-1.5.2, FR-1.5.3)
- Ingest pipeline populates `quarantined_tests` from test entries in artifact
  JSON (test_id, name, issue_url, quarantined_at, last_run_status)
- Responsive layout via CSS (no build step)

**MUST NOT implement:**
- Manual quarantine/unquarantine from UI (read-only in v1)
- Docker image (M8)
- Authentication beyond network-level
- Export or API endpoints
- Background polling daemon (M8)

## Acceptance criteria

From [milestones.md](index.md#m7-dashboard-analytics-and-ui):

1. Project detail page shows quarantined tests with issue links. (FR-1.5.1)
2. Trend charts render correctly with at least 7 days of data.
3. Filters narrow the displayed data correctly (date range, status, search all compose).
4. Search returns matching tests by name or `test_id`.
5. Cross-repo overview page shows aggregated statistics. (FR-1.5.5)
6. Dashboard ingests and displays data from real CLI output (not just fixtures). (FR-1.5.2)
7. UI is responsive on desktop and tablet viewports. (FR-1.5.6)
8. Unit and integration tests cover: chart data queries, filter logic, search,
   cross-repo aggregation.
9. On-demand sync fires on page load when data is stale (`last_pulled_at` is null or
   older than 5 minutes), reads `QUARANTINE_GITHUB_TOKEN` or `GITHUB_TOKEN` from env,
   and respects the debounce (no second call within 5 minutes). (FR-1.5.3, NFR-2.3.4)
10. Artifact ingestion populates `quarantined_tests`: new entries get `quarantined_at`
    from the artifact timestamp; subsequent ingestion updates `last_run_status` but
    preserves `quarantined_at`. (FR-1.5.1, FR-1.5.4)

## Verification

**Flow correctness:** No sequence diagram applies. Verification is
data-correctness focused:
- Trend chart queries MUST aggregate by date, one data point per day
- Cross-repo aggregation MUST sum across all repos in `dashboard.yml`
- Search MUST be case-insensitive and match partial test names
- Filters MUST compose (project + date range + status applied together)
- Quarantine test list MUST link to the correct GitHub Issue URL
- MUST validate that real CLI output (from M5) ingests and displays correctly
- Trend charts MUST handle sparse data (days with no runs)
- On-demand sync MUST read token from `QUARANTINE_GITHUB_TOKEN` (falls back to `GITHUB_TOKEN`)
- On-demand sync MUST fire on first page load (`last_pulled_at` is null)
- On-demand sync MUST NOT fire if `last_pulled_at` is within the last 5 minutes (debounce)
- `ingestArtifact` MUST upsert `quarantined_tests` rows for every test entry with `status: "quarantined"`, deriving `last_run_status` from `original_status` (`"failed"` → `"failing"`, `"passed"` → `"passing"`, `null` → `null`)
- `ingestArtifact` MUST preserve `quarantined_at` on conflict (INSERT OR IGNORE semantics for that field)
- `issue_url` MUST be constructed as `https://github.com/{owner}/{repo}/issues/{issue_number}` when `issue_number` is non-null

**Requirements:** The implementation MUST satisfy FR-1.5.1, FR-1.5.2, FR-1.5.3, FR-1.5.5 in
[functional-requirements.md](../specs/functional-requirements.md) and
NFR-2.1.2, NFR-2.3.4 in
[non-functional-requirements.md](../specs/non-functional-requirements.md).

**Build:** `make dash-test && make dash-lint` MUST pass.
