# M4: Quarantine State + Exclusion

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**MUST implement:**
- Read `quarantine.json` from `quarantine/state` branch via Contents API
- Write updated `quarantine.json` with SHA-based CAS; retry up to 3 times on
  409 with re-read and merge (union, quarantine wins)
- Batch check issue status via GitHub Search API; unquarantine tests whose
  issues are closed
- Framework-specific test exclusion flags (per cli-spec)
- GitHub Actions cache fallback for degraded mode
- Degraded mode: all tests run when state unavailable, warnings to stderr
- `--strict`, `--dry-run`, `--exclude PATTERN` flags
- Rate limit header tracking, warn below 10%

**MUST NOT implement:**
- GitHub Issue creation (flaky tests added to state but no issue created)
- PR comments or `--pr` flag

## Acceptance criteria

From [milestones.md](../planning/milestones.md#m4-quarantine-state--exclusion):

1. `quarantine.json` reflects newly detected flaky tests after a run. (FR-1.2.1)
2. Quarantined tests are excluded from execution. (FR-1.2.2, FR-1.2.3)
3. Closing a quarantine issue unquarantines the test. (FR-1.2.4)
4. CAS conflict merge produces the union of quarantined tests. (FR-1.2.5)
5. Degraded mode: API unreachable → all tests run, exit based on results
   only. (FR-1.6.2, FR-1.6.3)
6. `--strict`: infrastructure errors MUST cause exit 2.
7. `--dry-run`: no state changes written.
8. `--exclude`: matching tests ignored by quarantine.
9. Unit tests cover: state read/write/merge, CAS retry, exclusion flags,
   exclude patterns, degraded mode, exit codes with quarantine state.
10. E2E test: full run flow against a real GitHub repo. (see `e2e/run.test.js`)

Requirements: [functional](../planning/functional-requirements.md),
[non-functional](../planning/non-functional-requirements.md).
