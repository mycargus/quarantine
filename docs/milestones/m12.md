# M12: Dashboard App Auth Foundation

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**MUST implement:**
- Add Contract 19 and Contract 20 endpoint definitions to `schemas/github-api.json`
- Contract tests in `test/contract/github-app.test.js` (3 endpoints, 5 response cases)
- Pure function `generateJWT(clientID, privateKeyPEM, now)` → `dashboard/app/lib/jwt.server.ts`
- `InstallationTokenProvider` with token exchange, caching, and proactive refresh → `dashboard/app/lib/installation-token.server.ts`
- GitHub HTTP client wrapper with `X-RateLimit-Remaining` monitoring (NFR-2.8.3): read `X-RateLimit-Remaining` and `X-RateLimit-Limit` from every response; log warning at < 20% remaining → `dashboard/app/lib/github-client.server.ts`
- Unit tests for `generateJWT` (colocated `.test.ts`); interface tests for token exchange → `dashboard/test/installation-token.integration.test.ts`

**Required environment variables (dashboard reads these at runtime):**
- `QUARANTINE_APP_CLIENT_ID` — JWT `iss` claim (GitHub recommends Client ID per official docs)
- `QUARANTINE_APP_PRIVATE_KEY` — PEM-encoded RSA private key (value)
- `QUARANTINE_APP_PRIVATE_KEY_PATH` — path to PEM file (alternative to above)

**MUST NOT implement:**
- OAuth login, session management, or auth routes (M13)
- `installations` table, installation discovery, or background sync (M14)
- `source: github-app` config mode or `shouldSyncInstallations` (M14)
- E2E tests requiring real App credentials (M16)
- CLI changes of any kind (ADR-026 — no CLI code changes in v2)

**Operational prerequisite:** The dev GitHub App must be registered (M16 one-time setup step 1) before local development or manual testing. Integration tests use mock servers and do not require real App credentials.

## Acceptance criteria

From [test-strategy.md](../specs/test-strategy.md) and [contracts.md](../specs/contracts.md):

1. `generateJWT(clientID, privateKeyPEM, now)` produces an RS256 JWT with `iss`=clientID, `iat`=now−60s, `exp`=now+9min. Test key pairs MUST be generated programmatically — no fixture keys committed. (FR-1.14.1, NFR-2.7.2)
2. Invalid PEM input, a public key, or empty `clientID` each throw a descriptive error identifying the problem. (FR-1.14.1)
3. `InstallationTokenProvider` exchanges a JWT for an installation token via `POST /app/installations/{id}/access_tokens` with **no** `permissions` body parameter. (FR-1.14.2, FR-1.14.3)
4. Cached token is reused on subsequent calls; mock server call counter MUST remain 1 after two provider calls within the token's validity window. (FR-1.14.4, NFR-2.8.1)
5. Token is refreshed proactively when < 5 min remain before expiry; at most one `POST /access_tokens` call per ~55 min per installation. (FR-1.14.4, NFR-2.8.1)
6. Network error during token exchange: provider logs a warning and returns an error. MUST NOT throw or crash the process. (FR-1.14.5)
7. Private key MUST be read from `QUARANTINE_APP_PRIVATE_KEY_PATH` or `QUARANTINE_APP_PRIVATE_KEY` only. Client ID from `QUARANTINE_APP_CLIENT_ID`. MUST NOT be hardcoded in source or config. (NFR-2.7.1, FR-1.13.4)
8. GitHub HTTP client monitors `X-RateLimit-Remaining` and `X-RateLimit-Limit` on every response. Logs warning when remaining < 20% of limit. Unit tests MUST cover: warning logged when below threshold, no warning when above threshold, and missing headers do not cause an error. (NFR-2.8.3)
9. Contract tests: `POST /app/installations/{id}/access_tokens` → 201/401/403/404/422; `GET /app/installations` → 200; `GET /installation/repositories` → 200. All offline via Prism. The 403 case covers suspended installations or insufficient permissions. The 422 case covers validation failures (e.g., spammed endpoint). (Contract 19, Contract 20)
10. `make dash-test && make contract-test` MUST pass.

**Concurrency note:** If two callers request a token simultaneously when the cache is expired or near-expiry, at most one token exchange should occur. The implementation should use a pending-promise pattern or mutex to coalesce concurrent requests. (NFR-2.8.1, Scenario 44)

## Verification

**Build:** `make dash-test && make contract-test` MUST pass.

**JWT invariants (per [NFR-2.7.2](../specs/non-functional-requirements.md#27-app-auth-security)):**
- `alg` MUST be `RS256`. `iat` MUST equal `now − 60s`. `exp` MUST equal `now + 9min`.
- `iss` MUST equal the clientID argument exactly.
- MUST NOT use a hardcoded private key in tests; key pairs MUST be generated programmatically.

**Rate limit monitoring invariants (per [NFR-2.8.3](../specs/non-functional-requirements.md#28-app-auth-performance)):**
- Every GitHub API response MUST be checked for `X-RateLimit-Remaining` and `X-RateLimit-Limit` headers.
- When `remaining / limit < 0.20`, a warning MUST be logged with the resource, remaining count, and reset time.
- Missing headers MUST NOT cause an error (some endpoints may not return them).

**Token provider invariants (per [sequence diagram 9](../specs/sequence-diagrams.md#9-dashboard-jwt--installation-token-exchange-v2)):**
- `POST /app/installations/{id}/access_tokens` request body MUST NOT contain a `permissions` field.
- After 2 provider calls within the validity window, mock server MUST have received exactly 1 request.
- After simulated expiry, the next provider call MUST trigger exactly 1 new exchange request.
