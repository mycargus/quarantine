# M17: CLI v2 Read-Only Mode

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**MUST implement:**
- Two-step token resolution: `QUARANTINE_GITHUB_TOKEN` -> `GITHUB_TOKEN` -> exit 2
- v2 write skip: when `QUARANTINE_GITHUB_TOKEN` is NOT set, skip all GitHub write API calls (state update, issue creation, PR comment). Write results to disk only. Log a note directing the user to upload the artifact (with doc link, suppressed by `--quiet`).
- `cli_mode` field in results JSON: `"v1"` when PAT is used, `"v2"` when GITHUB_TOKEN is used. Dashboard uses this to know whether writes were already performed.
- v2 state branch missing (404): CLI proceeds with empty state, runs all tests with no exclusions. Does NOT exit 2. The dashboard creates the state branch on first artifact processing.
- Fork PR detection via `GITHUB_EVENT_PATH`: check both `pull_request` and `pull_request_target` events; head repo != base repo -> skip quarantine, exec raw command (SEC-7)
- Same-repo PR: `pull_request` or `pull_request_target` where head == base repo -> quarantine runs normally (SEC-7)
- Event payload parsing hardening: malformed/missing `GITHUB_EVENT_PATH` -> skip detection, quarantine runs normally (SEC-9)
- Context-aware missing-token error messages (GHA vs non-GHA) (UX-6)
- Mode-aware 403 error messages on state read (v1 vs v2) (UX-1)
- Exit 2 on no token (not degraded mode) (ERR-1)
- Verbose logging: `--verbose` shows which token path was taken
- Spec updates: `error-handling.md` (exit 2 for no-token), `cli-spec.md` (v2 token resolution, v2 write behavior, v2 state-branch-missing handling) — *verified by doc review*
- Doc updates: CI integration guide with zero-config workflow snippet — *verified by doc review*

**MUST NOT implement:**
- Dashboard write processing (M18)
- E2E round-trip testing (M19)
- Any dashboard communication from the CLI (no proxy, no OIDC, no env vars for dashboard URL)

## Acceptance criteria

From [plan: adr-036-token-proxy.md#m17-cli-v2-read-only-mode](../plans/adr-036-token-proxy.md#m17-cli-v2-read-only-mode):

1. `GITHUB_TOKEN` is used by default when no other token is configured (R-1)
2. Token resolution follows two-step order: `QUARANTINE_GITHUB_TOKEN` -> `GITHUB_TOKEN` -> exit 2 (R-2)
3. When `QUARANTINE_GITHUB_TOKEN` is NOT set, CLI makes zero GitHub write API calls: no state updates, no issue creation, no PR comments (R-3)
4. `.quarantine/results.json` written to disk with full data needed for dashboard processing, including `cli_mode` field (R-5)
5. CLI logs artifact upload note in v2 mode with doc link; suppressed by `--quiet` (R-3)
6. Fork PR detected (`pull_request` event, head repo != base repo) -> quarantine skipped, raw test command executed (SEC-7)
7. Fork detection also checks `pull_request_target` event name (SEC-7)
8. Same-repo PR (`pull_request` or `pull_request_target`, head == base repo) -> quarantine runs normally (SEC-7)
9. Malformed/missing `GITHUB_EVENT_PATH` -> fork detection skipped, quarantine runs normally (SEC-9)
10. No token available -> exit 2 with context-aware message (ERR-1, UX-6):
    - GHA: `"No GitHub token available. Ensure 'actions/checkout' runs before 'quarantine run'. If the problem persists, set QUARANTINE_GITHUB_TOKEN."`
    - Non-GHA: `"No GitHub token found. Set QUARANTINE_GITHUB_TOKEN or GITHUB_TOKEN."`
11. 403 on state branch read -> mode-aware actionable error (UX-1):
    - v1 (PAT): `"QUARANTINE_GITHUB_TOKEN lacks read access to the state branch (403). Check the token has 'contents: read' scope."`
    - v2 (GITHUB_TOKEN): `"Cannot read quarantine state (403). Check for SAML SSO enforcement or IP allowlist restrictions on this repository."`
12. v1 PAT mode: `QUARANTINE_GITHUB_TOKEN` set -> CLI does all reads AND writes (backward compatible, no behavioral change)
13. v2 state branch missing (404) -> CLI proceeds with empty state, runs all tests with no exclusions (R-6)
14. `--verbose` shows token resolution path (R-2)
15. `make cli-build && make cli-test && make cli-lint` pass
16. Spec and doc updates verified by doc review

## Verification

**v1-to-v2 migration:** When a user removes `QUARANTINE_GITHUB_TOKEN` from their
CI workflow (migrating from v1 to v2), the CLI MUST seamlessly switch to v2 mode:
read-only via `GITHUB_TOKEN`, results written to disk, no GitHub write API calls.
This is implicitly covered by the token resolution logic and write-skip behavior
but should be verified as an end-to-end scenario.

**Token resolution:** The implementation MUST resolve tokens in this exact sequence:
1. `QUARANTINE_GITHUB_TOKEN` set -> v1 mode (CLI reads and writes)
2. `GITHUB_TOKEN` set -> v2 mode (CLI reads only, writes results to disk)
3. No token found -> exit 2 with context-aware message

**v2 write skip:** When in v2 mode, the following functions MUST NOT be called:
- `WriteStateWithCAS` (state update)
- `CreateIssue` / issue dedup search (issue creation)
- `PostOrUpdateComment` / comment list (PR comment)

Verify by confirming zero GitHub write API calls are made during a v2-mode run.

**v2 state branch missing:** When the state branch returns 404 in v2 mode,
the CLI MUST proceed with empty state (no exclusions), NOT exit 2.

**Fork PR detection:** MUST check BOTH event names:
- `GITHUB_EVENT_NAME=pull_request` + `head.repo.full_name != base.repo.full_name`
- `GITHUB_EVENT_NAME=pull_request_target` + `head.repo.full_name != base.repo.full_name`

Same-repo PRs (head == base) MUST run quarantine normally.

**Build:** `make cli-build && make cli-test && make cli-lint` MUST pass.
