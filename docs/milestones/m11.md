# M11: Init UX Improvements

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**Before coding, run:**
- `/create-adr` for auto-detection design decisions (scope, API, vitest-over-jest priority)
- `/create-adr` for non-interactive init (scope change from v2, flag semantics, TTY policy)
- `/create-user-scenario` for all scenarios listed in the [plans](#specifications)

**MUST implement:**
- New `cli/internal/detect` package: `Scan(dir string) Result` (never returns error)
- Detection from `package.json` (`dependencies` + `devDependencies`) and `Gemfile`
- Modified `quarantine init`: single detected framework → pre-fills one suite entry
  with defaults; multiple detected → pre-fills one suite entry per detected framework;
  none detected → writes commented example suite entry. Detection is advisory only —
  no interactive framework prompt or numbered list (multi-framework selection deferred
  with M9; see [multi-suite plan](../plans/multi-suite-support.md#d4-quarantine-init-creates-config-with-pre-filled-suites-user-edits-yaml)).
- Flags on `quarantine init`: `--yes`/`-y`, `--retries`, `--junitxml`
- `--yes` mode: skips prompts, overwrites config with log
- Retries range validation (1–10) in both interactive and non-interactive paths
- TTY warning when stdin is not a terminal and `--yes` is absent

**MUST NOT implement:**
- Package manager detection (no consumer in v1)
- JUnit XML path detection from existing config files
- `--config` flag on `quarantine init` (removed per ADR-010 amendment)
- `--framework` flag (framework field removed per ADR-030)
- Multi-framework numbered-list selection prompt (Case B) — deferred with M9;
  detection now pre-fills multiple suite entries, not a selection prompt
- Changes to `quarantine run`, `quarantine doctor`, or `quarantine version`

## Acceptance criteria

From [index.md](index.md#m11-init-ux-improvements):

1. `quarantine init` detects jest/vitest from `package.json` and pre-fills a
   suite entry with `junitxml` and `rerun_command` defaults in config. (FR-1.1.6)
2. `quarantine init` detects rspec from `Gemfile` and pre-fills a suite entry
   with rspec defaults in config. (FR-1.1.6)
3. Multiple detected frameworks: pre-fills one suite entry per framework; no
   numbered-list prompt. (FR-1.1.6)
4. No detection → writes commented example suite entry; no prompt for framework.
5. Malformed `package.json` → silent fallback to no-detection behavior.
6. `quarantine init --yes` creates config without prompts, exits 0.
7. `quarantine init --yes --retries 11` exits 2 with actionable error.
8. Interactive retries prompt rejects out-of-range values and re-prompts.
9. `make cli-build && make cli-test && make cli-lint` pass.

Requirements: [functional](../specs/functional-requirements.md),
[non-functional](../specs/non-functional-requirements.md).

## Verification

**Build:** `make cli-build && make cli-test && make cli-lint` MUST pass.

**Detect package:** `detect.Scan()` MUST never return an error (advisory-only API).
Detection failures MUST silently fall back to Case C (no-detection prompt).

**Non-interactive:** `quarantine init --yes` without `--framework` MUST exit 2.
Flag overrides MUST skip exactly the flagged prompts and no others.

**Retries:** Both interactive and non-interactive paths MUST reject values outside
1–10 with a clear error or re-prompt.

**Requirements:** Implementation MUST satisfy [FR-1.4.2](../specs/functional-requirements.md).
