# M6: Dashboard Scaffolding + Data Ingestion

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**MUST implement:**
- React Router v7 app scaffolding: Vite, TypeScript, Tailwind CSS, Biome, pnpm
- `dashboard/` directory: `app/routes/`, `app/lib/`, `app/components/`
- `dashboard.yml` config: `source: manual`, repo list, poll interval
- SQLite schema and migrations (per architecture.md §5.3); WAL mode
- Artifact polling pipeline: ETag conditional requests, zip extraction,
  JSON schema validation, SQLite upsert (keyed by `run_id`)
- Circuit breaker: 3 consecutive failures → 30-minute pause per repo
- Debounced on-demand pull: max 1 per repo per 5 minutes
- Basic project listing page: repo names, test run count, last sync timestamp
- Makefile targets: `dash-build`, `dash-test`, `dash-lint`
- Developed against golden test fixtures from `testdata/expected/`

**MUST NOT implement:**
- Analytics charts, trend visualization, quarantined test list
- Filters, search, cross-repo overview
- Docker image, authentication beyond network-level

## Acceptance criteria

From [milestones.md](../planning/milestones.md#m6-dashboard-scaffolding--data-ingestion):

1. `pnpm dev` starts the dashboard with a project listing page. (FR-1.5.6)
2. `pnpm build` produces a production build.
3. SQLite schema created via migrations on first run. (FR-1.5.4)
4. Artifact polling ingests golden fixtures into SQLite correctly. (FR-1.5.2)
5. Project listing shows repo names, test run counts, last sync time.
6. `dashboard.yml` with `source: manual` is parsed and repos configured.
7. ETag-based conditional requests avoid re-downloading unchanged data.
8. Circuit breaker pauses polling after 3 consecutive failures. (NFR-2.3.5)
9. Malformed artifact JSON skipped with warning (not a crash). (NFR-2.2.3)
10. Unit tests cover: artifact parsing, SQLite upsert, config parsing,
    circuit breaker.
11. Contract tests: fixture JSON validates against shared schema.

[non-functional](../planning/non-functional-requirements.md).

## Verification

**Flow correctness:** The implementation MUST match
[sequence-diagrams.md#7](../specs/sequence-diagrams.md#7-dashboard-artifact-ingestion).
Specifically:
- Polling MUST use ETag-based conditional requests (304 = skip)
- Artifact filter MUST match `quarantine-result` name prefix
- Downloads MUST follow 302 redirect to blob storage
- JSON MUST be validated against schema before SQLite upsert
- Upsert MUST be keyed by `run_id` for idempotency
- Circuit breaker MUST trigger on 3 consecutive failures, pause 30 min
- On-demand pull MUST be debounced: max 1 per repo per 5 min

**Requirements:** The implementation MUST satisfy FR-1.5.2, FR-1.5.3,
FR-1.5.4, FR-1.5.6, FR-1.11.2 in
[functional-requirements.md](../planning/functional-requirements.md) and
NFR-2.1.2, NFR-2.2.3, NFR-2.3.4, NFR-2.3.5, NFR-2.4.1, NFR-2.4.2 in
[non-functional-requirements.md](../planning/non-functional-requirements.md).

**Build:** `make dash-build && make dash-test && make dash-lint` MUST pass.

**E2E gap (deferred):** The dashboard's GitHub API interactions (`listArtifacts`
with ETag, `downloadAndExtractJson` with redirect/zip) are tested via injected
fetch mocks but lack E2E coverage against the real API. Adding E2E tests
requires a GitHub Actions workflow in `mycargus/quarantine-test-fixture` that
uploads a `quarantine-results` artifact so the test has something to list and
download. Once that workflow exists and has run at least once, use
`/create-e2e-test` to add `test/e2e/github-artifacts.test.js`.
