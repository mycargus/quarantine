## Scope

**MUST implement:**
- Artifact polling via `InstallationTokenProvider` (M12) when `source: github-app`. This requires refactoring the existing polling code in `github.server.ts` to route between installation tokens and PATs per project based on `installation_id`.
- Mixed auth: App-discovered repos use installation tokens; manual-mode repos continue using PATs
- Add `GET /user/installations/{installation_id}/repositories` endpoint definition to `schemas/github-api.json` (Contract 22)
- Contract test for `GET /user/installations/{installation_id}/repositories` → 200 response in `test/contract/github-app.test.js`
- User permission filtering: `GET /user/installations` (metadata only, no repos), then `GET /user/installations/{id}/repositories` per installation with logged-in user's access token (M13), intersected with App-discovered projects (M14). All paginated calls use `per_page=100` and follow GitHub `Link` header `rel="next"` to fetch all pages (reuses `parseLinkHeader` from M14). See [updated diagram 12](../specs/sequence-diagrams.md#12-dashboard-user-permission-filtering-v2).
- Pure function `filterProjectsByUserAccess(allProjects, userRepoIds)` → `dashboard/app/lib/permissions.server.ts` with colocated unit tests (`.test.ts`). Follows FC/IS principle per test strategy §1.
- User permission caching: cache `GET /user/installations` + per-installation repo results for 5 minutes per user (consistent with artifact polling debounce). TTL-based with lazy refresh: when a cached entry expires, the next request blocks while fetching fresh data (no background revalidation). Cache MUST have a maximum size (e.g., 1,000 entries) with LRU eviction to prevent unbounded memory growth.
- Empty-list handling: user with no accessible repos sees an empty project list, not an error
- Token refresh event logging with timestamp and new expiry
- Unit tests for `filterProjectsByUserAccess` (colocated `.test.ts`); interface tests → `dashboard/test/user-permissions.integration.test.ts`

**MUST NOT implement:**
- OAuth login, session management, or auth routes (M13, already implemented)
- JWT generation or `InstallationTokenProvider` (M12, already implemented)
- Installation discovery, sync loop, or schema migrations (M14, already implemented)
- E2E tests with real App credentials (M16)

## Acceptance criteria

From [test-strategy.md](../specs/test-strategy.md) and [functional-requirements.md](../specs/functional-requirements.md#115-dashboard-oauth):

1. Artifact polling uses `InstallationTokenProvider` tokens when `source: github-app`; manual-mode repos continue using PATs unchanged. Existing PAT-based polling tests MUST continue passing. (FR-1.15.5, FR-1.16.3, NFR-2.9.3)
2. Logged-in user sees only repos accessible via `GET /user/installations` then `GET /user/installations/{id}/repositories` per installation, intersected with App-discovered projects. (FR-1.15.4)
3. `filterProjectsByUserAccess(allProjects, userRepoIds)` is a pure function: returns the intersection as an array. Empty input → empty output. (FR-1.15.4)
4. User with no accessible repos sees an empty project list, not an error. (FR-1.15.4)
5. Invalid/expired user access token on `GET /user/installations`: returns 401 (cookie session has expired or token is invalid). (FR-1.15.2)
6. User permission results cached for 5 min per user. Within the cache window, no additional `GET /user/installations` calls. (NFR-2.8.3)
7. Installation token refresh events (from `InstallationTokenProvider`, not user sessions) logged with timestamp and new expiry. (NFR-2.10.1)
8. Contract tests: `GET /user/installations/{installation_id}/repositories` → 200/403/404. The 200 case validates `total_count` + `repositories` array. The 403 case covers a user without access to the installation (e.g., removed from org between calls). The 404 case covers an installation that was deleted between calls. All offline via Prism. (Contract 22)
9. `make dash-test && make contract-test` MUST pass.
10. All paginated API calls (`GET /user/installations`, `GET /user/installations/{id}/repositories`) use `per_page=100` and follow `Link` header `rel="next"` until no `next` link is present. Reuses `parseLinkHeader` from M14. Interface tests MUST verify multi-page filtering with a mock server returning 2+ pages. (FR-1.15.4)

## Verification

**Build:** `make dash-test` MUST pass.

**Permission filtering invariants (per [FR-1.15.4](../specs/functional-requirements.md#115-dashboard-oauth), [diagram 12](../specs/sequence-diagrams.md#12-dashboard-user-permission-filtering-v2)):**
- Dashboard MUST call `GET /user/installations` with the user's own access token (not the App's installation token).
- Dashboard MUST then call `GET /user/installations/{id}/repositories` per installation to get the user's accessible repos. `GET /user/installations` does NOT return repo lists (per GitHub API docs).
- `filterProjectsByUserAccess` MUST be a pure function: takes all App-discovered projects and the user's accessible repo IDs, returns the intersection.
- Displayed projects MUST be the intersection of App-discovered projects and repos the user can access.
- Zero accessible repos MUST produce an empty list, not a 403 or error page.

**Caching invariant:**
- User permission results (installations + repos) MUST be cached for 5 min per user via TTL with lazy refresh (no background revalidation — when the cache entry expires, the next request blocks while fetching fresh data).
- Within the cache window, repeated page loads MUST NOT trigger additional `GET /user/installations` calls.

**Mixed auth invariant (per [NFR-2.9.3](../specs/non-functional-requirements.md#29-app-auth-reliability)):**
- A dashboard running `source: github-app` with some pre-existing `source: manual` projects MUST poll manual projects using PATs and App projects using installation tokens in the same polling cycle.

**Pagination invariant (per [FR-1.15.4](../specs/functional-requirements.md#115-dashboard-oauth)):**
- All paginated API calls MUST use `per_page=100` and follow the `Link` header `rel="next"` URL until no `next` link is present. Reuses `parseLinkHeader` from M14.
- Interface tests MUST verify pagination with a mock server returning 2+ pages for the user permission flow.

**Refactoring risk:** This milestone modifies the existing artifact polling code in `github.server.ts` to support mixed auth. Existing PAT-based polling behavior MUST be preserved — all pre-existing dashboard tests MUST continue passing after the refactoring.
