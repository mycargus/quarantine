# M19: v2 Server-Side Writes E2E

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**MUST implement:**
- `test/e2e/v2-server-side-writes.test.js`: E2E tests observing **pre-seeded fixture data** from the fixture repo (`mycargus/quarantine-app-test-fixture`), which runs CI periodically with deliberately flaky tests. The dashboard has already processed these runs. E2E tests observe the outcomes without waiting for real-time processing.
- Verify v2 flow: dashboard-created state updates, issues, and PR comments with App bot identity
- Verify v1 backward compatibility: CLI-created issues and comments with PAT owner identity
- Verify CLI exit code correctness: fixture CI runs in v2 mode exit 0 (all tests pass) or exit 1 (test failures) regardless of dashboard processing state
- E2E tests MUST observe CI output -- MUST NOT run the CLI binary directly or pre-arrange GitHub state
- Review [E2E test conventions](../../test/e2e/README.md#philosophy) before writing any tests

**MUST NOT implement:**
- CLI code changes (complete in M17)
- Dashboard code changes (complete in M18)
- New scenario files in `docs/scenarios/` (E2E tests are not scenario-driven)

**Dependencies:** M17 (CLI v2 read-only mode) and M18 (dashboard write processing) MUST be complete and verified before M19 implementation begins.

## Acceptance criteria

From [plan: adr-036-token-proxy.md#m19-v2-server-side-writes-e2e](../plans/adr-036-token-proxy.md#m19-v2-server-side-writes-e2e):

1. E2E: fixture repo has dashboard-processed v2 runs; state branch contains at least one quarantined test entry written by the dashboard; issues created by the App bot identity (verify `creator.type == "Bot"`); PR comments contain suite-specific `<!-- quarantine:<suite> -->` markers
2. E2E: fixture repo has v1 PAT runs; CLI-created issues have PAT owner identity (`creator.type == "User"`); CLI-created comments have PAT owner identity (backward compat)
3. E2E: fixture repo v2 CI workflow runs completed with expected conclusion (`success` for exit 0, `failure` for exit 1) regardless of dashboard processing state; verified via `GET /repos/.../actions/runs` API
4. E2E: `make e2e-test` passes (existing E2E tests unbroken)

Requirements: [ADR-036 amendment](../adr/036-v2-cli-auth-github-token-default.md#amendment-2026-04-20-server-side-writes-replace-token-proxy).

## Acceptance scenarios

E2E tests for M19 do not use `docs/scenarios/` files. Tests live in
`test/e2e/v2-server-side-writes.test.js` and observe pre-seeded fixture data:

- **v2 flow:** Observe that the fixture repo's state branch contains quarantined tests written by the dashboard. Observe that issues have the App's bot identity as creator. Observe that PR comments use suite-specific `<!-- quarantine:<suite> -->` markers.
- **v1 backward compat:** Observe that fixture runs using `QUARANTINE_GITHUB_TOKEN` produced CLI-created issues and comments with the PAT owner identity (not the App bot).
- **CLI exit code:** Observe that fixture CI workflow runs in v2 mode completed with the expected exit code (check workflow run conclusion: `success` for exit 0, `failure` for exit 1).

## Verification

**E2E invariants (from [test/e2e/README.md](../../test/e2e/README.md)):**
- Tests MUST observe fixture CI output -- MUST NOT run the CLI binary, create fake JUnit XML, or pre-arrange GitHub state
- v2 flow test MUST verify App bot identity on issues and PR comments
- v1 backward compat test MUST verify PAT owner identity on issues and comments
- Exit code test MUST verify workflow run conclusion matches expected outcome
- All tests MUST be independent -- no shared state or ordering dependency
- Tests observe **pre-seeded data** from fixture runs that the dashboard has already processed -- no real-time waiting required

**E2E environment:**
- Tests MUST throw a descriptive error if fixture env vars are absent (following existing E2E pattern)
- Fixture repo MUST have both v2 runs (GITHUB_TOKEN only) and v1 runs (QUARANTINE_GITHUB_TOKEN) available for observation
- E2E job MUST gate execution to trusted branches only

**Build:** Existing `make e2e-test` and `make e2e-app-test` MUST remain passing. New E2E tests added to an appropriate CI job.
