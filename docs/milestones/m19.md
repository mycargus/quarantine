# M19: v2 Server-Side Writes E2E

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)

5. Use `/mikey:testify` to validate test quality.

## Scope

**Test approach:** Spawn the dashboard with `source: github-app` mode (using M16 App credentials) so it discovers and monitors both fixture repos via the App installation. Trigger sync via `GET /` (same pattern as `test/e2e/dashboard-sync.test.js`). Observe outcomes by querying the GitHub API and the dashboard's HTML pages.

**MUST implement:**
- `test/e2e/v2-server-side-writes.test.js`: E2E tests observing **pre-seeded fixture data** from two fixture repos. The App is installed on both, so the dashboard discovers and monitors both:
  - `mycargus/quarantine-app-test-fixture`: v2 workflow (GITHUB_TOKEN only, uploads artifact). Dashboard performs all writes using App installation tokens.
  - `mycargus/quarantine-test-fixture`: existing v1 workflow (PAT, CLI writes, uploads artifact). Dashboard ingests artifact but skips writes because `cli_mode == "v1"`.
- Verify v2 flow: dashboard-created state updates (Git commit author check on `quarantine/state` branch), issues with App bot identity (`creator.type == "Bot"`), PR comments with suite marker AND `user.type == "Bot"`
- Verify v1 `cli_mode` skip with two-part assertion:
  - **Positive ingestion signal:** dashboard's `GET /projects/{v1_owner}/{v1_repo}` page renders the "Showing X of Y quarantined tests" phrase with non-zero values, proving the v1 artifact was ingested into SQLite
  - **Negative skip signal:** zero issues with `creator.type == "Bot"` exist on the v1 fixture; zero PR comments have `user.type == "Bot"`
- Verify CLI exit code correctness: at least one recent v2 workflow run on the app fixture has `conclusion == "success"` (proves quarantine exclusion worked); workflow conclusion is independent of dashboard processing state
- E2E tests MUST observe CI output -- MUST NOT run the CLI binary directly or pre-arrange GitHub state
- Review [E2E test conventions](../../test/e2e/README.md#philosophy) before writing any tests

**MUST NOT implement:**
- CLI code changes (complete in M17)
- Dashboard code changes (complete in M18)
- New scenario files in `docs/scenarios/` (E2E tests are not scenario-driven)
- Pre-seeding the fixture data (operational step, not a test deliverable)

**Dependencies:**
- M17 (CLI v2 read-only mode) and M18 (dashboard write processing) MUST be complete and verified
- Pre-seeding (one-time operational step) MUST be complete:
  1. After M17 release, both fixtures' workflows pull `latest` on next daily cron — `cli_mode` appears in their artifacts
  2. Trigger `workflow_dispatch` on both fixtures to seed at least one M17-era run each. Wait for M18 dashboard to poll and ingest both
  3. Verify pre-seeding: app fixture has App-bot-authored issues + state branch commits; v1 fixture has CLI-authored (`creator.type == "User"`) issues only AND dashboard's project page shows test runs for it

## Acceptance criteria

From [plan: adr-036-token-proxy.md#m19-v2-server-side-writes-e2e](../plans/adr-036-token-proxy.md#m19-v2-server-side-writes-e2e):

1. E2E: at least one recent v2 run on `quarantine-app-test-fixture` produced dashboard-written artifacts:
   - State branch: at least one commit on `quarantine/state` authored by the App bot (Git commit author email contains `[bot]`); per-suite state file `.quarantine/<suite>/state.json` contains at least one quarantined test entry
   - Issues: at least one open issue with `quarantine` label has `creator.type == "Bot"` and `creator.login` matches the App's bot name (e.g., `quarantine-ci[bot]`)
   - PR comments: at least one comment on the v2 fixture's PR proxy has `<!-- quarantine:<suite> -->` marker AND `user.type == "Bot"`
2. E2E: `quarantine-test-fixture` (v1) verification (cli_mode skip), two-part:
   - Positive ingestion signal: `GET /projects/{v1_owner}/{v1_repo}` on a freshly-spawned dashboard renders "Showing X of Y quarantined tests" with non-zero values (proves dashboard saw the artifact)
   - Negative skip signal: ZERO open or closed issues on the v1 fixture have `creator.type == "Bot"`; ZERO comments on the v1 fixture's PR proxy have `user.type == "Bot"`; all CLI-created issues/comments have `creator.type == "User"`
3. E2E: at least one recent v2 workflow run on `quarantine-app-test-fixture` has `conclusion == "success"` (proves quarantine exclusion worked when dashboard-written state was applied); workflow conclusion is independent of dashboard processing state (verified via `GET /repos/.../actions/runs` API)
4. E2E: existing `make e2e-test` (v1 fixture, single-token tests) continues to pass; new tests live under the `e2e-app` job (M16-introduced) and are gated by App credential env vars

## Acceptance scenarios

E2E tests for M19 do not use `docs/scenarios/` files. Tests live in
`test/e2e/v2-server-side-writes.test.js` and observe pre-seeded fixture data:

- **v2 flow:** Spawn dashboard with `source: github-app` mode. Trigger sync via `GET /`. Query GitHub API: state branch on app fixture has commits authored by App bot; quarantine-labeled issues exist with `creator.type == "Bot"` and the App's bot login; PR comment exists with `<!-- quarantine:<suite> -->` marker AND `user.type == "Bot"`.
- **v1 `cli_mode` skip:** Same dashboard spawn (already monitoring both via App installation). After sync, `GET /projects/{v1_owner}/{v1_repo}` renders "Showing X of Y quarantined tests" with non-zero values (positive ingestion signal). Then query GitHub API: ZERO issues with `creator.type == "Bot"`, ZERO PR comments with `user.type == "Bot"` on the v1 fixture (negative skip signal).
- **CLI exit code:** Query `GET /repos/{app_fixture}/actions/runs?event=schedule&per_page=10` and confirm at least one recent run has `conclusion == "success"` (proves quarantine exclusion via dashboard-written state worked).

## Verification

**E2E invariants (from [test/e2e/README.md](../../test/e2e/README.md)):**
- Tests MUST observe fixture CI output -- MUST NOT run the CLI binary, create fake JUnit XML, or pre-arrange GitHub state
- v2 flow test MUST verify App bot identity (`creator.type == "Bot"`, `user.type == "Bot"`) on issues, comments, AND state branch commits (via Git commit author)
- v1 cli_mode skip test MUST verify BOTH the positive ingestion signal (dashboard project page renders test runs) AND the negative skip signal (zero bot-authored content on v1 fixture)
- Exit code test MUST verify workflow run conclusion matches expected outcome
- All tests MUST be independent -- no shared state or ordering dependency
- Tests observe **pre-seeded data** from fixture runs that the dashboard has already processed -- no real-time waiting required

**E2E environment:**
- Tests MUST throw a descriptive error if any required env var is absent (following existing E2E pattern)
- App MUST be installed on both fixture repos (per `GITHUB_APP_SETUP.md` step 4)
- Dashboard MUST be spawnable in `source: github-app` mode using the M16 App credentials
- `QUARANTINE_GITHUB_TOKEN` PAT MUST have read access to both fixture repos
- `quarantine-app-test-fixture` MUST have v2 runs (GITHUB_TOKEN only, no `QUARANTINE_GITHUB_TOKEN` set) producing artifacts with `cli_mode: "v2"`
- `quarantine-test-fixture` MUST have post-M17 runs (PAT) producing artifacts with `cli_mode: "v1"`
- Pre-seeding (M17 CLI release + at least one workflow_dispatch on each fixture + dashboard ingestion) MUST be complete before tests are meaningful
- E2E job MUST gate execution to trusted branches only

**Build:** Existing `make e2e-test` (v1 fixture, single-token tests) MUST remain passing. New M19 tests run in the M16-introduced `e2e-app` job (which has App credentials). New tests MUST NOT be picked up by the existing `e2e` job (which would throw on missing App env vars).
