## Scope

**MUST implement:**
- `test/e2e/v2-auth-proxy.test.js`: three E2E tests observing real fixture CI output for round-trip proxy, unreachable dashboard fallback, and 404-repo fallback (E2E-1)
- Fixture repo setup: a CI run in `mycargus/quarantine-app-test-fixture` (or dedicated proxy fixture) that uses `QUARANTINE_DASHBOARD_URL` pointing to a real deployed dashboard; this run MUST produce artifacts showing `quarantine[bot]` identity
- `make e2e-proxy-test` target and `e2e-proxy` CI job (following the pattern of `e2e-app` from M16)
- E2E tests MUST observe CI output — MUST NOT run the CLI binary directly or pre-arrange GitHub state
- Review [E2E test conventions](../../test/e2e/README.md#philosophy) before writing any tests — the "observe, don't arrange" philosophy applies to all proxy tests

**MUST NOT implement:**
- CLI code changes (complete in M17)
- Dashboard code changes (complete in M18)
- New scenario files in `docs/scenarios/` (E2E tests are not scenario-driven)

**Dependencies:** M17 (CLI token resolution) and M18 (dashboard proxy) MUST be complete and verified before M19 implementation begins.

## Acceptance criteria

From [plan: adr-036-token-proxy.md#m19-v2-auth-e2e](../plans/adr-036-token-proxy.md#m19-v2-auth-e2e):

1. E2E test verifies full round-trip: CLI → proxy → installation token → API calls succeed with `quarantine[bot]` identity (E2E-1, FR-1.11.1 revised)
2. E2E test verifies fallback: dashboard unreachable → CLI uses `GITHUB_TOKEN` within timeout window (E2E-1, NFR-2.1.1)
3. E2E test verifies fallback: dashboard returns 404 (repo not in installation) → CLI uses `GITHUB_TOKEN` (E2E-1, NFR-2.1.1)

## Acceptance scenarios

E2E tests for M19 do not use `docs/scenarios/` files. Tests live in
`test/e2e/v2-auth-proxy.test.js` and observe real fixture CI output:

- **Round-trip:** Fixture CI runs with `QUARANTINE_DASHBOARD_URL` set. Test observes that resulting GitHub comments/issues have `quarantine[bot]` as actor (not the PAT user).
- **Unreachable fallback:** Fixture CI runs with `QUARANTINE_DASHBOARD_URL` pointing to an unreachable host. Test observes that quarantine still completes using `GITHUB_TOKEN`.
- **404 fallback:** Fixture CI runs against a repo not in the App installation. Test observes fallback to `GITHUB_TOKEN`.

## Verification

**E2E invariants (from [test/e2e/README.md](../../test/e2e/README.md)):**
- Tests MUST observe fixture CI output — MUST NOT run the CLI binary, create fake JUnit XML, or pre-arrange GitHub state
- Round-trip test MUST verify `quarantine[bot]` identity on resulting comments or issues (not just token receipt)
- Fallback tests MUST verify `GITHUB_TOKEN` was used by observing the actor identity on GitHub API responses
- Fallback timeout test MUST complete within the 3s proxy timeout window plus test overhead
- All three tests MUST be independent — no shared state or ordering dependency between them

**E2E environment:**
- Tests MUST throw a descriptive error if proxy fixture env vars are absent (following existing E2E pattern)
- `e2e-proxy` CI job MUST gate execution to trusted branches only
- Fixture repo MUST have the GitHub App installed and `QUARANTINE_DASHBOARD_URL` configured in its CI workflow
- Proxy fixture env vars MUST be independent of existing `e2e` and `e2e-app` job variables
- Proxy fixture env vars: `QUARANTINE_PROXY_TEST_OWNER`, `QUARANTINE_PROXY_TEST_REPO`, `QUARANTINE_DASHBOARD_URL` (at minimum)

**Build:** `make e2e-proxy-test` MUST pass on CI. Existing `make e2e-test` and `make e2e-app-test` MUST remain passing.
