# M14: E2E Integration + CI

## Workflow

1. Read this manifest to understand scope and constraints.
2. Load the [acceptance scenarios](#acceptance-scenarios) for the current command.
3. Use `/mikey:tdd <scenario-file>` to implement against those scenarios.
4. After implementation, verify against [acceptance criteria](#acceptance-criteria)
   and [flow verification](#verification).
5. Use `/mikey:testify` to validate test quality.

## Scope

**MUST implement:**
- One-time setup: register dev GitHub App (manual); create `mycargus/quarantine-app-test-fixture` (dedicated App test fixture with quarantine config, state branch, and GitHub workflows that upload quarantine result artifacts); install App on the fixture repo; create a dedicated GitHub test account and pre-authorize the dev App; configure CI secrets/variables (see [§5.4](../plans/github-app.md#54-e2e-tests)). OAuth callback URL for dev App: `http://localhost:3000/auth/github/callback`.
- E2E tests in `test/e2e/github-app-dashboard.test.js`: real token exchange, real discovery, real artifact polling
- Separate `e2e-app` CI job: runs App-based E2E tests (dashboard token exchange, discovery, artifact polling) and CLI E2E tests with App-generated tokens against `mycargus/quarantine-app-test-fixture`. Passes App credentials as env vars (including `QUARANTINE_APP_ORIGIN`). Throws if env vars are absent (matching existing E2E pattern). Gates execution to trusted branches only. This is independent of the existing `e2e` job (PAT-based, `mycargus/quarantine-test-fixture`).
- Documentation: recommended CI workflow snippet using `actions/create-github-app-token` to pass App token as `QUARANTINE_GITHUB_TOKEN`
- Verify fixture repo artifact data: confirm `mycargus/quarantine-app-test-fixture` has quarantine result artifacts (uploaded by the fixture repo's own GitHub workflows) so that E2E artifact polling tests (scenario 40) have data to download
- Playwright browser E2E test in `test/e2e-browser/dashboard-oauth.test.ts`: add `@playwright/test` as dev dependency at repo root. Navigates to `/auth/login`, completes GitHub OAuth with the dedicated test account (dev App pre-authorized, so consent page is skipped), verifies session cookie is set and authenticated page loads, then verifies logout clears the session. Test account MUST have TOTP 2FA enabled with the TOTP secret stored as a CI secret for Playwright to generate codes.
- Documentation: branch protection bypass instructions for repo admins

**MUST NOT implement:**
- Webhook handling (ADR-027 — deferred to v3)
- CLI code changes (ADR-026 — no CLI changes for v2)
- Production App registration — dev App only; prod registration happens at launch

## Acceptance criteria

From [github-app plan §2.1](../plans/github-app.md#21-app-registration-and-configuration) and [§5.4](../plans/github-app.md#54-e2e-tests):

1. Dev App registered under `mycargus` with permissions: `contents:read+write`, `issues:read+write`, `pull_requests:write`, `actions:read`. Webhooks MUST be disabled. OAuth callback URL registered. (FR-1.13.1, FR-1.13.2, FR-1.13.3)
2. Dev App private key NOT committed to source, config files, or `quarantine.yml`. Stored only in CI secret `QUARANTINE_APP_PRIVATE_KEY`. (FR-1.13.4, FR-1.13.5)
3. `mycargus/quarantine-app-test-fixture` created with quarantine config, `quarantine/state` branch, and GitHub workflows that upload quarantine result artifacts. Dev App installed on this fixture repo. Dedicated GitHub test account created with the dev App pre-authorized and TOTP 2FA enabled. All required CI secrets and variables configured per [§5.4](../plans/github-app.md#54-e2e-tests), including `QUARANTINE_APP_ORIGIN` and test account credentials. (§5.4 one-time setup)
4. E2E: dashboard generates JWT from dev credentials, calls `POST /app/installations/{id}/access_tokens`, receives a valid token, and the token succeeds for `GET /repos/{owner}/{repo}`. (§5.4)
5. E2E: `GET /app/installations` + `GET /installation/repositories` return the fixture repo (`quarantine-app-test-fixture`) with real credentials. (§5.4)
6. E2E: artifact polling with installation token behaves identically to existing PAT-based polling E2E tests. (§5.4)
7. App E2E tests require App credentials — throw a descriptive error if env vars are absent (matching existing E2E pattern in `test/e2e/`). `e2e-app` CI job gates execution to trusted branches only. (§5.4)
8. Separate `e2e-app` CI job added with App credentials env vars; runs App-based E2E tests and CLI E2E with App tokens against `mycargus/quarantine-app-test-fixture`. Only runs on trusted branches with secret access. Existing `e2e` job (PAT-based, `mycargus/quarantine-test-fixture`) unchanged. (§5.4)
9. CLI E2E with App token: existing CLI E2E tests MUST pass in the `e2e-app` job when `QUARANTINE_GITHUB_TOKEN` is set to a token generated by `actions/create-github-app-token` and the target repo is `mycargus/quarantine-app-test-fixture`. This validates the recommended CI integration path (ADR-026) end-to-end. (§5.4, ADR-026)
10. Setup documentation includes the recommended GitHub Actions workflow snippet using `actions/create-github-app-token` to pass an App-generated token as `QUARANTINE_GITHUB_TOKEN`. (FR-1.13.1, ADR-026)
11. Branch protection bypass instructions documented for repo admins: Settings > Rules > Rulesets > Bypass list > Add App. (FR-1.17.2)
12. `make e2e-test` (PAT-based, existing fixture) and `make e2e-app-test` (App-based, `quarantine-app-test-fixture`) MUST pass on CI. Existing E2E tests MUST remain passing.
13. Playwright OAuth E2E: browser navigates to `/auth/login`, completes GitHub OAuth with the dedicated test account (dev App pre-authorized, TOTP 2FA handled by Playwright), verifies authenticated page load, and logout clears the session. Test account credentials stored as CI secrets. `make e2e-browser-test` MUST pass on CI.

Requirements: [FR-1.13.x](../specs/functional-requirements.md#113-github-app-registration),
[FR-1.17.x](../specs/functional-requirements.md#117-branch-protection-bypass).

## Verification

**Build:** `make e2e-test` (PAT-based), `make e2e-app-test` (App-based), and `make e2e-browser-test` (Playwright) MUST pass on CI. Existing CLI E2E tests MUST remain passing.

**E2E invariants (per [§5.4](../plans/github-app.md#54-e2e-tests)):**
- Token exchange test MUST use real dev App credentials, not mocks.
- Fixture repo (`quarantine-app-test-fixture`) MUST appear in `GET /app/installations` + `GET /installation/repositories` response.
- When `QUARANTINE_APP_PRIVATE_KEY` (or related vars) is absent, App E2E tests MUST throw a descriptive error (matching existing E2E pattern). `e2e-app` job gates execution to trusted branches.
- Artifact polling via installation token MUST behave identically to existing PAT-based polling.
- CLI E2E with App token MUST run against `mycargus/quarantine-app-test-fixture`, not the existing `quarantine-test-fixture`.

**Browser E2E invariants:**
- Playwright test MUST complete the full OAuth flow against real GitHub: `/auth/login` → GitHub login page → TOTP 2FA → redirect back to dashboard callback → session cookie set.
- After login, the dashboard MUST return authenticated content (not 401).
- After `GET /auth/logout`, a subsequent request MUST return 401.
- Test account MUST have the dev App pre-authorized (consent page skipped) and TOTP 2FA enabled.

**Security invariant (per [FR-1.13.4](../specs/functional-requirements.md#113-github-app-registration)):**
- No PEM-formatted key material committed to source. Verify: `git grep -r -- '-----BEGIN' '*.ts' '*.js' '*.go' '*.yml' '*.yaml'` MUST return no matches.
