# M14: Installation Discovery

## Scope

**MUST implement:**
- SQLite migrations: `installations` table (with `suspended_at` and `removed_at` nullable timestamp columns — GitHub API has no `status` enum; suspension is conveyed via `suspended_at`, removal is inferred by sync diff); nullable `installation_id` FK on `projects`
- Pure function `shouldSyncInstallations(lastSyncedAt, now, intervalMs)` → `dashboard/app/lib/installation-sync.server.ts`
- `syncInstallations()`: calls `GET /app/installations` then `GET /installation/repositories` per installation, upserts both tables; MUST NOT throw. All paginated calls use `per_page=100` and follow GitHub `Link` header `rel="next"` to fetch all pages.
- Pure function `parseLinkHeader(header: string | null)` → `dashboard/app/lib/github-client.server.ts`: extracts `next` URL from GitHub `Link` header for pagination. Unit test colocated (`.test.ts`)
- `source: github-app` config validation in `dashboard/app/lib/config.server.ts`
- Startup credential validation: missing/invalid App credentials MUST fail before sync starts
- Startup sync: blocking until `syncInstallations()` completes before any HTTP traffic
- Background loop: `setInterval` at 15 min; first tick fires 15 min after startup
- Shutdown hooks: `SIGTERM` + `SIGINT` clear the interval. Accept an injectable `shutdownSignals` array (default `['SIGTERM', 'SIGINT']`) so interface tests can use `process.emit()` without risk to the test runner (see [testing approach](#signal-testing-approach) below).
- Startup sync timeout: if `syncInstallations()` does not complete within the configured timeout (default 60 seconds, configurable via `startupSyncTimeoutMs` parameter for testability), log an error and exit. The HTTP server MUST NOT start with incomplete data.
- Repo removal: set `installation_id` to NULL; preserve `test_runs` and `quarantined_tests`
- Interface tests → `dashboard/test/installation-sync.integration.test.ts` and `dashboard/test/installation-loop.integration.test.ts`

**MUST NOT implement:**
- Artifact polling wiring to `InstallationTokenProvider` (M15 — github-app mode integration)
- User permission filtering via `GET /user/installations` (M15)
- Mixed auth (manual + App) artifact polling (M15)
- E2E tests with real App credentials (M16)
- Webhook-based installation events (ADR-027 — webhooks deferred to v3)

## Acceptance criteria

From [test-strategy.md](../specs/test-strategy.md) and [functional-requirements.md](../specs/functional-requirements.md#116-installation-discovery):

1. `installations` table (with `suspended_at` and `removed_at` nullable timestamp columns) and nullable `installation_id` on `projects` created via migrations. `source: manual` rows have `installation_id` = NULL. (FR-1.16.1, FR-1.16.2)
2. `shouldSyncInstallations(lastSyncedAt, now, intervalMs)`: stale/null timestamp returns true; fresh timestamp returns false; exact boundary returns false. (FR-1.16.5)
3. `source: github-app` validates without `repos` array; `repos` present is silently ignored. `source: manual` without `repos` still returns a validation error. (FR-1.15.6, Contract 21)
4. Missing/invalid App credentials at startup: MUST fail with a descriptive error identifying the missing credential. HTTP server MUST NOT start. Discovery MUST NOT start. (NFR-2.9.1)
5. Startup with `source: github-app`: dashboard MUST NOT serve HTTP traffic until `syncInstallations()` completes. After startup, `installations` and `projects` tables are populated. (FR-1.16.4)
6. Background loop re-syncs every 15 min. First tick fires 15 min after startup, not immediately. (FR-1.16.5)
7. Shutdown: `SIGTERM` and `SIGINT` clear the interval; no sync fires after shutdown. (FR-1.16.6)
8. `syncInstallations()` catches all errors, logs them, MUST NOT throw. Existing table data unchanged after a failed sync. Loop continues on next interval. The entire upsert batch (all installations + projects) MUST be wrapped in a single SQLite transaction so that a mid-sync crash leaves the database in its pre-sync state, not partially updated. (FR-1.16.7, NFR-2.9.4)
9. Suspended installation: `installations.suspended_at` updated from the GitHub API response (`null` = active, ISO 8601 timestamp = suspended). Artifact polling MUST skip repos linked to suspended installations. (FR-1.16.8, NFR-2.9.2)
10. Repo removed from installation: project's `installation_id` set to NULL. Historical `test_runs` and `quarantined_tests` data MUST be preserved. (FR-1.16.10)
11. Discovery logs installations found, repos added/removed, errors. (NFR-2.10.2)
12. Startup sync timeout: if `syncInstallations()` exceeds the configured timeout (default 60 seconds), dashboard logs an error and exits with a non-zero code. HTTP server MUST NOT start. Interface tests use a short timeout (e.g., 100ms) with a slow mock server. (NFR-2.9.1)
13. Removed installation detection: installations that no longer appear in `GET /app/installations` (uninstalled) MUST be marked with a non-NULL `removed_at` timestamp. Artifact polling MUST skip repos linked to removed installations. Historical data (`test_runs`, `quarantined_tests`) MUST be preserved. If a previously removed installation reappears, `removed_at` MUST be set back to NULL and its repos MUST be re-linked to the installation (the upsert logic re-discovers repos and sets `installation_id` on matching projects). (FR-1.16.8, Scenario 46)
14. `make dash-test` MUST pass.
15. All paginated API calls (`GET /app/installations`, `GET /installation/repositories`) use `per_page=100` and follow `Link` header `rel="next"` until no `next` link is present. `parseLinkHeader(header)` is a pure function that extracts the `next` URL. Unit tests MUST cover: header with `next` and `last`, header with only `next`, null/empty header, malformed header. Interface tests MUST verify multi-page discovery with a mock server returning 2+ pages. (FR-1.16.4)

## Verification

**Build:** `make dash-test` MUST pass.

**Startup invariants (per [FR-1.16.4](../specs/functional-requirements.md#116-installation-discovery), [diagram 10](../specs/sequence-diagrams.md#10-dashboard-startup-sync--discovery-loop-v2)):**
- Credential validation MUST occur before `syncInstallations()`. Missing credentials → exit with error, server never starts.
- API call order MUST be: `GET /app/installations` (JWT auth), then `GET /installation/repositories` (installation token auth) per installation.
- HTTP traffic MUST NOT be served until both calls complete and tables are populated.
- If startup sync exceeds 60 seconds, the process MUST exit with a descriptive error. The HTTP server MUST NOT start with incomplete data.
- GitHub numeric `id` MUST be used for installations, repos, and users — never names or slugs. (FR-1.16.9)
- **Known limitation:** `GET /app/installations` uses JWT auth (max ~9 min lifetime per M12). For extremely large deployments where full sync exceeds 9 minutes, the JWT may expire mid-pagination of the installations list. This is acceptable because (a) the startup sync timeout (60s) would fire first, and (b) the 15-minute background loop generates a fresh JWT each cycle.

**Loop invariants (per [FR-1.16.5, FR-1.16.6](../specs/functional-requirements.md#116-installation-discovery)):**
- At t=0, sync runs once (startup). At t=15min, second sync runs. No sync fires between startup and first interval.
- After SIGTERM/SIGINT, sync MUST NOT be called again.

**Transaction invariant:**
- The entire upsert batch (all installations + all projects) MUST be wrapped in a single SQLite transaction (`BEGIN` / `COMMIT`). If any upsert fails or the process crashes mid-sync, the transaction rolls back and the database retains its pre-sync state.

**Error isolation invariant (per [NFR-2.9.4](../specs/non-functional-requirements.md#29-app-auth-reliability)):**
- A 500 from `GET /app/installations` MUST leave `projects` table unchanged and MUST NOT propagate an exception to the HTTP request handler.

**Suspension detection invariant (per [FR-1.16.8](../specs/functional-requirements.md#116-installation-discovery)):**
- The GitHub API has no `status` field on installations. Suspension is conveyed via `suspended_at` (null = active, ISO 8601 timestamp = suspended). The sync logic MUST read `suspended_at` from the API response and store it in `installations.suspended_at`.
- Installations that no longer appear in `GET /app/installations` (uninstalled) MUST be marked with a non-NULL `removed_at` timestamp. This is detected by comparing known installation IDs against the API response set.

**Data preservation invariant (per [FR-1.16.10](../specs/functional-requirements.md#116-installation-discovery)):**
- After repo removal, `SELECT count(*) FROM test_runs WHERE project_id = {removed}` MUST return the same count as before removal.

**Pagination invariant (per [FR-1.16.4](../specs/functional-requirements.md#116-installation-discovery)):**
- All paginated API calls MUST use `per_page=100` and follow the `Link` header `rel="next"` URL until no `next` link is present.
- `parseLinkHeader` MUST be a pure function with no side effects, tested via colocated unit tests.
- Interface tests MUST verify pagination with a mock server returning 2+ pages (e.g., first page returns 100 items with a `Link: <...?page=2>; rel="next"` header, second page returns remaining items with no `next` link).

### Signal testing approach

Use `process.emit('SIGTERM')` (in-process EventEmitter dispatch), the approach used by Lightship and other production Node.js libraries for Vitest-compatible signal handler testing. This is safe because:

1. `process.on('SIGTERM', handler)` registers on Node's EventEmitter. `process.emit('SIGTERM')` invokes it through the same API — no real OS signal is sent, so the test runner is not killed.
2. `process.kill(process.pid, 'SIGTERM')` sends a real OS signal that kills the Vitest runner (documented in Jest #10374, same risk in Vitest). Do not use this approach.
3. The shutdown handler should accept an injectable `terminate` function (default: `process.exit`) so tests can spy on it without exiting: `vi.spyOn(process, 'exit').mockImplementation(() => {})` or inject a mock.

**Test pattern:**
```typescript
// Start the discovery loop with short interval (100ms)
const { cleanup } = startDiscoveryLoop({ intervalMs: 100, shutdownSignals: ['SIGTERM'] })
// Verify sync fires at least once
// Emit signal
process.emit('SIGTERM')
// Wait one interval
// Assert: sync call count has NOT increased after signal
```
