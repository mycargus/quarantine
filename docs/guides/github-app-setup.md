# Setting Up the Quarantine GitHub App

This guide walks through registering a GitHub App for Quarantine, configuring
CI to use App-generated tokens, and setting up the dashboard for App-based
authentication.

**Who is this for?** Platform engineers and org admins setting up Quarantine
for their organization.

**Prerequisites:** Quarantine v1 is already working with PATs. You want to
upgrade to fine-grained permissions, higher rate limits, and org-wide
installation.

---

## Why a GitHub App?

PATs create friction that scales badly:

- Every repo needs manual PAT configuration
- Tokens are long-lived secrets tied to individual human accounts
- `GITHUB_TOKEN` in Actions is limited to 1,000 req/hr/repo
- Branch protection can block writes to `quarantine/state`
- PR comments and issues appear from a personal account

A GitHub App solves these problems. See [ADR-008](../adr/008-auth-strategy.md)
for the full rationale.

---

## 1. Register the App

Register a new GitHub App under your organization (or personal account for
dev/testing).

**Settings:**

| Setting | Value |
|---------|-------|
| App name | `quarantine` (globally unique, max 34 chars) |
| Homepage URL | Your dashboard URL or repo URL |
| Callback URL | `https://<dashboard-host>/auth/github/callback` |
| Webhooks | **Disabled** (uncheck "Active") |
| Token expiration | **Enabled** (Settings > Optional features) |

**Permissions (repository):**

| Permission | Access | Purpose |
|-----------|--------|---------|
| Contents | Read & write | Read/write `quarantine.json` on `quarantine/state` branch |
| Issues | Read & write | Create and search quarantine issues |
| Pull requests | Write | Post PR comment summaries |
| Actions | Read | Download test result artifacts (dashboard) |

After registration, note:

- **App ID** (numeric, e.g., `123456`) -- used by `actions/create-github-app-token`
- **Client ID** (string, e.g., `Iv23li...`) -- used as JWT `iss` claim and OAuth client ID
- **Client secret** -- used for OAuth token exchange

Generate a private key (PEM file) and store it securely. Never commit it to
source.

---

## 2. Install the App

Go to the App's installation page and install it on your organization. Choose
either:

- **All repositories** -- every repo can use Quarantine automatically
- **Selected repositories** -- pick specific repos

The dashboard discovers installed repos automatically via
`GET /app/installations` and `GET /installation/repositories`.

---

## 3. Configure CI

### Using `actions/create-github-app-token`

The CLI requires **no code changes** for App auth. The official
`actions/create-github-app-token` action generates a short-lived installation
token that the CLI consumes through the existing `QUARANTINE_GITHUB_TOKEN`
env var.

```yaml
- uses: actions/create-github-app-token@v3
  id: app-token
  with:
    app-id: ${{ vars.QUARANTINE_APP_ID }}
    private-key: ${{ secrets.QUARANTINE_APP_PRIVATE_KEY }}

- name: Run tests with quarantine
  run: quarantine run -- jest --ci --reporters=default --reporters=jest-junit
  env:
    QUARANTINE_GITHUB_TOKEN: ${{ steps.app-token.outputs.token }}

- name: Upload quarantine results
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: quarantine-results-${{ github.run_id }}
    path: .quarantine/results.json
```

### CI secrets and variables

Configure these on your repository (Settings > Secrets and variables > Actions):

| Type | Name | Value |
|------|------|-------|
| Secret | `QUARANTINE_APP_PRIVATE_KEY` | PEM file contents |
| Variable | `QUARANTINE_APP_ID` | App's numeric ID |

The token generated by the action expires after 1 hour and is automatically
revoked when the job completes.

### PAT fallback

Repos that haven't migrated to the App continue using PATs via
`QUARANTINE_GITHUB_TOKEN`. The CLI can't distinguish App tokens from PATs --
both are Bearer tokens. No migration cutover is required.

## 4. Configure the Dashboard

The dashboard needs its own credentials to generate installation tokens for
artifact polling and to handle OAuth login.

### Environment variables

| Env Var | Required | Description |
|---------|----------|-------------|
| `QUARANTINE_APP_CLIENT_ID` | Yes | App client ID. Used as JWT `iss` claim and OAuth client ID. |
| `QUARANTINE_APP_CLIENT_SECRET` | Yes | App client secret. Used for OAuth token exchange. |
| `QUARANTINE_APP_PRIVATE_KEY_PATH` | Yes* | Path to PEM file. |
| `QUARANTINE_APP_PRIVATE_KEY` | Yes* | PEM contents as env var value. Alternative to path. |
| `QUARANTINE_APP_ORIGIN` | Yes | Dashboard origin for OAuth redirect URI (e.g., `https://dashboard.example.com`). |

\* One of `QUARANTINE_APP_PRIVATE_KEY_PATH` or `QUARANTINE_APP_PRIVATE_KEY`
is required.

### dashboard.yml

Switch from manual mode to App mode:

```yaml
# Before (v1): manual repo list with PAT
source: manual
repos:
  - owner: my-org
    repo: my-project

# After (v2): repos discovered from App installations
source: github-app
```

When `source: github-app`, the `repos` array is silently ignored if present.
See the [Dashboard Deployment Guide](dashboard-deployment.md) for full
configuration details.

## 5. Branch Protection Bypass

Branch protection bypass is a **repo-admin action**, not an App capability.
The App's `contents:write` permission makes it eligible, but a repo admin must
explicitly add it.

**Steps:**

1. Go to **Settings > Rules > Rulesets** for the repository
2. Select the ruleset protecting the `quarantine/state` branch
3. Under **Bypass list**, click **Add bypass**
4. Select the Quarantine App
5. Save

When bypassed, the CLI's writes to `quarantine/state` succeed without the
Actions cache fallback. When NOT bypassed, existing fallback behavior
continues to work.

## GitHub Platform Constraints

Reference these when designing or debugging your integration:

| Constraint | Value | Impact |
|-----------|-------|--------|
| JWT `exp` max from `iat` | 10 minutes | JWTs regenerated per token request (or cached ~9 min). |
| Installation token expiry | 1 hour | Token provider must track expiry and refresh proactively. |
| User access token expiry | 8 hours (opt-in per App) | Session cookie `Max-Age: 28800`. No refresh tokens. |
| Max private keys per App | 25 | Supports rotation with headroom. Keys don't expire. |
| Installation token rate limit | 5,000 base, max 12,500 req/hr | Scales +50/hr per repo beyond 20, +50/hr per user beyond 20. |
| Secondary rate limits | 100 concurrent, 900 pts/min | Dashboard must stagger API calls. |
| Content creation | 80/min, 500/hr | Limits state writes. Not a concern at expected volume. |
| OAuth token exchange URL | `github.com` (not `api.github.com`) | Different base URL. Handled by `@remix-run/auth`. |
| Branch protection bypass | Repo-admin configured | App cannot self-grant bypass. |
