# ADR-021: Forward-Compatible Configuration Fields

**Status:** Accepted
**Date:** 2026-03-17

## Context

Quarantine's v1 config (`quarantine.yml`) supports a narrow set of integrations: GitHub Issues for issue tracking, `[quarantine]` as the only label, and GitHub PR comments as the only notification channel. v2 plans to expand these to support Jira, custom labels, Slack notifications, and more.

If v1 omits these fields entirely and v2 introduces them as new schema additions, users upgrading to v2 face a config schema change. If the fields are added with different structures or semantics than v1 users expect, it creates confusion. Additionally, the `version: 1` field alone does not communicate what expansions are coming.

## Decision

Include v2-targeted fields in the v1 config schema with restricted allowed values. The CLI validates these fields and rejects unsupported values with clear error messages.

The pattern:

1. **Field exists in v1 schema** with a default value that reflects the only supported option.
2. **CLI validates** the field against a restricted set of values for the current version.
3. **Error messages are specific:** e.g., `"issue_tracker: jira is not supported in v1. Supported values: github"`.
4. **v2 expands the allowed values** without changing the field name, type, or location in the schema.

Fields using this pattern in v1:

| Field | v1 Allowed Values | v1 Default | v2+ Expansion |
|-------|-------------------|------------|---------------|
| `issue_tracker` | `github` | `github` | `jira`, `linear`, etc. |
| `labels` | `[quarantine]` | `[quarantine]` | Any custom labels |
| `notifications` | `{ github_pr_comment: true/false }` | `{ github_pr_comment: true }` | `slack`, `email`, etc. |

## Alternatives Considered

- **Omit v2 fields entirely from v1 schema:** Simpler v1 config, but v2 must introduce new fields. Users upgrading need to learn new config options. Risk of v2 field names/structures conflicting with what users might have assumed. Rejected because it defers pain to the upgrade path.
- **Accept any value but ignore unsupported ones silently:** Low friction, but users may not realize their config is being ignored. A user who sets `issue_tracker: jira` and sees no Jira tickets would be confused. Rejected for poor feedback.
- **Use `version: 2` to gate new values:** Requires users to change the version field to access new features, which is a reasonable upgrade step. However, it does not prevent config schema surprises -- the field structure could still change. This pattern is complementary (v2 uses `version: 2` AND expands allowed values) rather than an alternative.

## Consequences

- (+) Zero breaking config changes between v1 and v2. Users upgrading to v2 expand allowed values in existing fields rather than learning new fields.
- (+) Clear error messages for unsupported values serve as documentation of what v2 will support.
- (+) Config file structure is self-documenting -- users can see what integrations exist even if they are not yet supported.
- (+) JSON Schema validation (`quarantine-config.schema.json`) can use `enum` constraints that are loosened in v2 without structural changes.
- (-) v1 config appears to support features it does not. Mitigated by validation errors and comments in the default config generated by `quarantine init`.
- (-) Slightly more complex v1 validation logic (must check allowed values per version). Mitigated by the simplicity of the restricted value sets.
- (-) Users may attempt to use v2 features and be frustrated by rejection. Mitigated by error messages that explicitly name the feature as v2 and suggest upgrading.
